Grundlagen

Architekturzustände (3 Stück) ▾

📍 Baseline Architecture

IST-Architektur

Aktueller Zustand der Architektur — Ausgangspunkt aller Transformationen.

🎯 Target Architecture

ZIEL-Architektur

Gewünschter zukünftiger Zustand — das angestrebte Ergebnis.

🔀 Transition Architecture — Zwischenzustände auf dem Weg von Baseline zu Target Architecture. Hilft, die Transformation in machbare Schritte zu unterteilen.
Architektur-Roadmap ▾
Beschreibt den Übergang von IST- zur ZIEL-Architektur.
Enthält:
  • Konkrete Maßnahmen
  • Zeitpläne
  • Verantwortlichkeiten
  • Sequenz von Transformationen

"Manage Complexity" — Komplexität reduzieren und kontrollieren

Business Architecture

Capability (Geschäftsfähigkeit) ▾
Definition: Eine Fähigkeit eines Unternehmens zur Erbringung einer bestimmten Leistung.
Abgrenzung vom Geschäftsprozess
CapabilityGeschäftsprozess
WAS — stabile FähigkeitWIE — Abfolge von Aktivitäten
Stabil, unabhängig von der Org-StrukturKann sich ändern (Reorganisation etc.)
z.B. "Kundenverwaltung"z.B. "Kunde anlegen → prüfen → bestätigen"
Eigenschaften einer Capability Map (mind. 4) ▾
Prüfungsrelevant: Mindestens 4 Eigenschaften nennen können!
  • Geschäftsorientiert — nicht IT- oder prozessorientiert
  • Stabil — unabhängig von Organisationsstruktur und Prozessen
  • Hierarchisch aufgebaut — Ebene 1, Ebene 2, ...
  • Redundanzfrei — jede Fähigkeit nur einmal
  • Vollständig — alle relevanten Fähigkeiten abgedeckt
Value Stream (Wertstrom) ▾
Definition: End-to-End-Darstellung der Werterbringung für einen Stakeholder.
Eigenschaften
  • Fokussiert auf Nutzen (nicht auf interne Prozesse)
  • Orientiert am Ergebnis — Verb-Noun-Syntax (z.B. "Deliver Order")
  • Keine klassische Prozessdarstellung
  • Bildet alle möglichen Szenarien ab

Warum Business Architecture Silodenken überwindet (3 Argumente) ▾
  • Gemeinsame Sprache über Abteilungsgrenzen hinweg
  • Ganzheitliche Betrachtung der gesamten Unternehmensleistung
  • Transparenz über Abhängigkeiten zwischen verschiedenen Bereichen

Architekturprinzipien

Definition & 3 Bestandteile (Metamodell nach Aier et al.) ▾
Qualitative Leitlinien, die die Gestaltung der Unternehmensarchitektur steuern. Sie wirken als "guide rails", nicht als strikte Gesetze.
3 Bestandteile
BestandteilBedeutung
StatementDie eigentliche Aussage des Prinzips
RationaleBegründung — Warum gilt dieses Prinzip?
ImplicationsKonsequenzen — Was bedeutet das konkret?
4 Treiber / Einflussfaktoren nach Greefhorst & Proper (2011) ▾
  • Ziele & Strategien ("goals & objectives")
  • Werte ("values")
  • Probleme ("issues")
  • Rahmenbedingungen ("constraints")
Weitere Treiber: Risiken und Chancen
5 Anforderungen nach Sandkuhl et al. (2015) ▾
  • Zielorientierung
  • Aussagekräftige Beschreibung
  • Sorgfältige Kommunikation
  • Prozessverankerung
  • Messung des Erfüllungsgrads
SMART-Kriterien für Architekturprinzipien (Greefhorst & Proper) ▾
BuchstabeEnglischBedeutung
SSpecificSpezifisch formuliert
MMeasurableMessbar / überprüfbar
AAchievableErreichbar / umsetzbar
RRelevantRelevant für das Unternehmen
TTime framedZeitlich begrenzt / gültig
Beispiel-Beurteilungen: Ist es ein Architekturprinzip? ▾
Prüfungstipp: Prinzip = qualitative Leitlinie. KEIN Prinzip: technische Standards, Definitionen, Methodiken oder konkrete Toolentscheidungen.
✅ JA"Applikationen sollen modular aufgebaut sein" — allgemeine Design-Richtlinie
❌ NEIN"Google Chrome ist unser Standard-Webbrowser" — technische Vorgabe / Standard
❌ NEIN"IT-Lösungen werden anhand der Capability Map fachlich eingeordnet" — Methodik, kein Prinzip
✅ JA"Die Architektur sollte minimale Komplexität aufweisen"
✅ JA"Kaufsoftware wird bevorzugt gegenüber Eigenentwicklung"
❌ NEIN"Microsoft SQL Server ist unsere Standard-Datenbank-Technologie" — Standard, kein Prinzip
❌ NEIN"Ein Wertstrom ist eine Ende-zu-Ende-Darstellung" — Definition, kein Prinzip
✅ JA"Applikationen müssen Cloud-fähig sein"
❌ NEIN"Apache Kafka ist unsere Standard-Event-Streaming-Plattform" — Standard, kein Prinzip

EA-Modellierung (ArchiMate)

Was ist ArchiMate? ▾
Standardisierte Modellierungssprache zur visuellen Darstellung von Unternehmensarchitekturen — ein Open Group Standard.
4 Ebenen (Layers) des ArchiMate Full Frameworks ▾
LayerFokus
Strategy LayerStrategische Ziele, Fähigkeiten, Ressourcen
Business LayerGeschäftsprozesse, Rollen, Akteure, Services
Application LayerAnwendungen, Softwarekomponenten, Daten
Technology LayerInfrastruktur, Hardware, Netzwerk
Weitere Layers: Physical Layer und Implementation & Migration Layer
3 Aspekte (pro Layer) ▾
AspektKernfrageBeispiele
Active StructureWer führt aus?Business Actor, Application Component
BehaviorWas wird getan?Business Process, Application Function
Passive StructureWomit / Worauf?Business Object, Data Object

  • Business Service

  • Contract

💻 Application Layer

  • Application Component
  • Application Function
  • Application Service
  • Data Object

🖥️ Technology Layer

  • Node
  • System Software
  • Technology Service
  • Artifact

Viewpoint, View & Concern ▾
BegriffDefinition
ViewpointStandardisierte Perspektive / Vorlage (Template) auf die Architektur
ViewKonkrete Darstellung einer Architektur auf Basis eines Viewpoints
ConcernAnliegen eines Stakeholders, das durch eine View adressiert wird
Merkhilfe: Viewpoint = Template (abstrakt)  |  View = Instanz (konkret)

6 EAM-Gestaltungsdimensionen ▾
#DimensionKernfrage
1EAM-MotivationWarum wird EAM eingeführt? Ziele, Mission, Vision
2EAM-AngebotWelche Services / Leistungen bietet EAM? (Service Portfolio)
3EAM-InhalteWas wird erfasst? Metamodell der Architekturinhalte

3 Organisationsmodelle ▾

🏛️ Zentralisiert

Eine zentrale EA-Abteilung steuert alle Aktivitäten.

Geeignet für stark zentralisierte Unternehmen

🔀 Dezentralisiert

Jede Business-Unit hat eigene EA-Experten.

Geeignet für autonome Divisionen

⚖️ Hybrid ★

Zentrale Steuerung + dezentrale Umsetzung.

Häufigster Ansatz in der Praxis
